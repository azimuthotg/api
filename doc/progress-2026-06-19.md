# Progress 2026-06-19 — Monitor: ติดตามวันหมดอายุ JWT token

## ที่มา (ปัญหาจริง)
API ออก JWT ให้ระบบปลายทางใช้ (access อายุ **365 วัน**) แต่เป็น stateless — ไม่มี record
ว่าใครได้ token ไปเมื่อไหร่/หมดเมื่อไหร่ เมื่อวาน (2026-06-18) **2 ระบบล่ม**เพราะ token หมดอายุ
ใช้เวลานานกว่าจะรู้สาเหตุว่าเป็นเพราะ token หมด

## สิ่งที่ทำ (2 ฟีเจอร์ใน /monitor/)

### 1. Token Inspector — `/monitor/token/`
วาง JWT (access/refresh) → ตรวจ signature ด้วย `SECRET_KEY` (HS256) แต่ **ไม่เช็ค exp**
(จึงดู token ที่หมดไปแล้วได้) → แสดง token_type / user_id / iat / exp / jti / เหลือกี่วัน
+ แถบสถานะ หมดแล้ว(แดง)/ใกล้หมด ≤30 วัน(เหลือง)/ปกติ(เขียว). ใช้วินิจฉัยเร็วเวลาระบบล่ม.

### 2. Token issuance log + dashboard — `/monitor/tokens/`
- model `TokenIssueLog` (table `token_issue_log`, migration `0006`) — event(obtain/refresh),
  username, user_id, jti, issued_at, expires_at, ip, user_agent.
- ครอบ endpoint ออก token: `apiapp/token_views.py` →
  `LoggingTokenObtainPairView` / `LoggingTokenRefreshView` (subclass ของ simplejwt,
  log หลังออกสำเร็จ, ถ้า log พังไม่กระทบ auth). wire ใน `apiproject/urls.py` แทนของเดิม
  (URL/พฤติกรรมเดิมทุกอย่าง).
- dashboard สรุป **token ล่าสุดของแต่ละ user** เรียงใกล้หมดก่อน + ไฮไลต์สี
  + การ์ดสรุป (จำนวน user, หมดแล้ว, ใกล้หมด ≤30 วัน) + ประวัติการออกทั้งหมด (ค้น username + แบ่งหน้า).

## โครงสร้างไฟล์
- `apiapp/token_utils.py` — `decode_token()` (PyJWT, verify sig ไม่ verify exp), `log_token_issue()`.
- `apiapp/token_views.py` — logging token views + `_username_from_access()`.
- `apiapp/monitor_views.py` — `monitor_token_inspect`, `monitor_token_log`, `_token_status()` (เกณฑ์ `TOKEN_SOON_DAYS=30`).
- templates `monitor/token_inspect.html`, `monitor/token_log.html`. ปุ่มลิงก์เพิ่มใน dashboard เดิม.
- `apiapp/admin.py` — register `TokenIssueLog`.

## ข้อจำกัด (สำคัญ)
log เริ่มเก็บ**เฉพาะ token ที่ออกหลังจาก deploy นี้** — ระบบที่ถือ token เก่าอยู่แล้ว
(ออกก่อนหน้า) จะยังไม่โผล่ใน dashboard จนกว่าจะ re-login. ช่องว่างนี้ใช้ **Token Inspector**
อุดได้ทันที (เอา token ปัจจุบันของระบบนั้นมาวางดูวันหมด).

## เทสต์ที่ทำ (local, ยังไม่แตะ prod DB)
- `manage.py check` ผ่าน, `makemigrations` → `0006_tokenissuelog` สร้างแล้ว.
- decode_token round-trip: valid/expired/tampered ถูกต้อง; `_token_status` คืน soon/expired ถูก.
- ทั้งสอง template compile ผ่าน.

## ค้าง / Deploy (ฟีเจอร์ token)
commit `ca41bd9` push ขึ้น origin/main แล้ว แต่**ยังไม่ deploy** (รวบไป deploy พร้อมฟีเจอร์ถัดไป).
Deploy ผ่าน `deploy.ps1` จะ `migrate 0006` สร้างตาราง `token_issue_log` ให้เอง.

# ฟีเจอร์ 2 — Monitor: log การใช้งาน endpoint ตรวจสอบ/ดึงข้อมูล นศ.-บุคลากร

## ที่มา (ปัญหาจริง)
หลายระบบภายนอกเรียก endpoint ตรวจสอบ (login ผ่าน AD) / ดึงข้อมูล นศ.-บุคลากร แต่ผู้ดูแล
**มองไม่เห็นเลยว่าระบบไหนเรียกเข้ามา ผลเป็นยังไง** เวลาผู้ใช้ถามว่า "login จากระบบ X เข้าไม่ได้
เพราะอะไร" ก็สืบไม่ได้. เดิม log แค่ `auth_ldap` → BindingLog (เพื่อ Monitor การผูก LINE) และ
ไม่เก็บตัวตนของระบบที่เรียก; ส่วน `auth_and_get_student/personnel` + endpoint ดึงข้อมูลไม่ log เลย.

## การตัดสินใจ (ยืนยันกับเจ้าของงาน)
- แต่ละระบบมีบัญชี Django แยก → ใช้ `client_user` (username ของ JWT) ระบุระบบได้แม่น.
- ครอบทั้ง auth + endpoint ดึงข้อมูล (student/personnel GET).

## สิ่งที่ทำ
- model `ApiAccessLog` (table `api_access_log`, migration `0007`) — created_at, client_user,
  client_ip, user_agent, api_version, endpoint, method, target_user (รหัส นศ./บุคลากร),
  http_status, result(success/fail), reason_code, message, duration_ms. **ไม่เก็บรหัสผ่าน.**
- `apiapp/access_log.py` — `ApiAccessLogMixin` ดักที่ `finalize_response` (จุดเดียวครอบได้ทั้ง
  custom action + list/retrieve) + `log_api_access()` (กันพัง, ไม่กระทบ flow). target_user เดาจาก
  body `userLdap` → lookup kwarg → query param. reason จาก `request._api_access_reason` (AD) ก่อน,
  ไม่งั้นจาก HTTP status + `detail` ใน body.
- ครอบ mixin เข้า: v2 `StudentsInfoViewSetV2` / `StaffInfoViewSetV2` / `LDAPAuthViewSetV2`,
  v1 `StudentsInfoViewset` / `StaffInfoViewSet` / `LDAPAuthViewSet`. auth_ldap (v1+v2) ส่งสาเหตุ AD
  เข้า access log ด้วย (1 บรรทัด `request._api_access_reason = ...`).
- หน้า Monitor `/monitor/api-usage/` (`monitor_api_usage` + `templates/monitor/api_usage.html`):
  การ์ดสรุปวันนี้ (เรียก/ล้มเหลว/จำนวนระบบ) + ตารางภาพรวมต่อระบบ 7 วัน (เรียก/ล้มเหลว/%)
  + ตัวกรอง (ค้นรหัส นศ./ระบบ/IP/endpoint, เลือกระบบ, ผล, เวอร์ชัน) + รายการ + แบ่งหน้า.
  ปุ่มลิงก์เพิ่มในหน้า Monitor เดิม. register admin แล้ว.

## เทสต์ (local, ไม่แตะ prod DB)
`check` ผ่าน, `makemigrations` → `0007_apiaccesslog`. helper (`_reason_from_status`,
`_target_user` ทั้ง retrieve/body/search, `_short_message` จาก body + AD meta) ถูกต้อง. template compile ผ่าน.

## หมายเหตุ / follow-up
- log จะเริ่มเก็บเฉพาะ request หลัง deploy. endpoint ดึงข้อมูลอาจ volume สูง → อนาคตอาจต้องมี
  นโยบาย retention/ลบ log เก่า (ยังไม่ทำ).

## Deploy (รวมทั้ง 2 ฟีเจอร์)
